# RAFIQ Import Prototype And Raw Landing-Zone Design

Status: Day 7 Design Finalized  
Design date: 2026-06-14

## Purpose

The import prototype must load every technically usable audited resource into
the complete private platform without changing raw source evidence or
prematurely forcing unlike source structures into one canonical model.

```text
source
  -> acquisition snapshot
  -> immutable raw objects
  -> parser/import run
  -> source-shaped staging records
  -> validation findings and quality flags
  -> candidate canonical mappings
  -> private publication
  -> separately governed public publication
```

## Core Principles

1. Raw bytes are immutable.
2. A source, acquisition snapshot, and file/object are different identities.
3. Staging preserves source structure before canonical transformation.
4. Every staged row links to its raw object, import run, and source locator.
5. Corrections occur only in versioned derived transformations.
6. Duplicates, conflicts, blanks, malformed fields, and gaps are evidence.
7. Private import approval and public-release approval are separate states.
8. Re-running the same parser against the same object is idempotent.
9. Replacing one source version must not change unrelated canonical identity.
10. Client applications never access raw, ingest, or staging schemas directly.

## Three-Level Source Identity

### Source

The continuing provider/resource identity, for example:

- `qul-quran-script-uthmani-88`
- `tanzil-translation-en-sahih`
- `fawaz-hadith-api`
- `semakhadis`

### Acquisition Snapshot

One dated, version-specific acquisition of a source. It records:

- branch, tag, commit, release, resource version, or API date
- acquisition method
- acquisition timestamp
- terms and attribution captured at that time
- aggregate file count and byte count
- snapshot-level technical and approval status

### Raw Object

One immutable file, API response, archive member, database, or generated
payload inside the snapshot. It records:

- relative path or storage key
- byte length
- SHA-256 checksum
- media type, encoding, and format
- archive/member relationship where applicable
- principal, mirror, generated, evidence, or metadata role

This model supports both paired QUL JSON/SQLite exports and the Fawaz snapshot
containing more than 650,000 files.

## Landing-Zone Layout

New acquisitions should use:

```text
data/
  raw/
    <domain>/
      <provider>/
        <source_id>/
          <snapshot_id>/
            payload/
            evidence/
            metadata/
  manifests/
    sources/
    snapshots/
    objects/
  checksums/
    <snapshot_id>.sha256.csv
  staging_reports/
    <domain>/
      <import_run_id>/
```

Recommended `snapshot_id`:

```text
YYYY-MM-DD_<version-or-commit-short>
```

Existing Day 2-Day 6 paths remain valid and must not be moved merely to match
the new layout. Register them as legacy raw-object paths. Future acquisitions
use the versioned layout.

## Raw Object Roles

- `principal`: authoritative bulk payload selected for parsing
- `mirror`: equivalent compact/minified/alternate serialization
- `generated`: per-record or derived export generated by the distributor
- `metadata`: source metadata, catalog, schema, or README
- `evidence`: terms, copyright, attribution, or source page
- `archive`: downloaded ZIP/TAR/GZIP container retained when required
- `support`: fonts, code, seed support files, or ancillary assets

Object role does not imply religious or editorial authority.

## Immutability Rules

- Never edit, normalize, re-encode, deduplicate, unzip-overwrite, or repair a
  registered raw object in place.
- A changed byte sequence is a new raw object with a new checksum.
- Archive extraction records parent archive checksum and member path.
- Repository snapshots retain commit identity and repository metadata.
- Failed or malformed objects remain registered with failure findings.
- Raw deletion requires an explicit retention decision; a successful import is
  not sufficient reason to delete evidence.

## Import Run Contract

Every parser execution creates an `import_run` containing:

- source snapshot
- parser name and version
- code revision
- configuration hash
- start/end timestamps
- status
- input object count
- parsed, rejected, warning, and staged record counts

The idempotency key is:

```text
snapshot_id + parser_name + parser_version + configuration_hash
```

A rerun with changed parser logic creates a new run. It does not overwrite the
previous run or its findings.

## Row-Level Lineage

Every staging row must retain:

- `source_id`
- `snapshot_id`
- `raw_object_id`
- `import_run_id`
- `source_record_key`
- source row number, JSON pointer, SQL key, or archive member
- raw record JSON where practical
- raw and normalized hashes where applicable

For a derived record based on multiple source rows, use a lineage relation
containing every parent row and transformation event.

## Quality And Correction Model

Quality flags are additive records, not destructive status columns. Required
fields:

- domain
- finding code
- severity
- affected staging record or raw object
- source-observed or RAFIQ-derived classification
- description
- evidence
- resolution state
- transformation event where corrected in staging

Examples:

- `blank_text`
- `exact_duplicate`
- `coverage_gap`
- `malformed_keyword`
- `invalid_utf8`
- `trailing_comma_json`
- `unreliable_grade_field`
- `grade_disagreement_review`
- `missing_attribution`
- `rights_unknown`

## Approval And Environment Scope

Approval state belongs to a source snapshot and may be inherited by staged and
derived records. Required independent dimensions:

- technical validation
- rights
- attribution
- editorial review
- scholar/content review
- publication state

Private mode uses every technically usable record. Public-release mode exposes
only records whose exact source versions pass the required approval gates.

Derived records inherit the strictest public-release restriction of all parent
records.

## Security Boundary

For PostgreSQL/Supabase:

- `ingest` schema: source registry, snapshots, objects, runs, findings
- `staging` schema: source-shaped parsed records and candidate mappings
- canonical content schema: application-owned normalized records
- public-release views/API: approved records only

`ingest` and `staging` are private schemas. Revoke access from `anon` and
`authenticated`. Only server-side importer and approved admin services may
read or write them.
